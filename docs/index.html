claude-session — functions

A bash script. Its shell functions are documented below.

usage(exit_code = 0)

Prints the header comment block (lines 2–30 of the script) as help text by stripping the leading #, then exits with exit_code. Invoked by -h and on any option-parsing error.

Args: $1 — exit status (default 0).

in_tmux()

Predicate: returns success when the script is running inside an existing tmux client, detected via the $TMUX environment variable. Used to decide between attach-session (from a bare shell) and switch-client (from within tmux, where attach would nest).

Returns: exit 0 if inside tmux, else 1.

attach_or_switch(target)

Attaches the current terminal to tmux session target, or, if already inside tmux, switches the active client to it (avoids the "sessions should be nested with care" error).

Args: $1 — target session name.

restore-claude — functions

build()

Creates the six-window claude session: planner, security, mail-bot (each running claude --continue) and lab-1/lab-2/lab-3 (each running claude --resume so you pick the conversation), with the correct starting directory per window. If the session already exists the script attaches instead of calling this.

Note: must be run from a real terminal/SSH shell, not a Claude Code tool, so the tmux server lives in the login session and persists.

install.sh — behaviour

install.sh [--no-boot]

Idempotent installer. Resolves its own repo location, then: (1) symlinks cj, claude-session, restore-claude into ~/.local/bin; (2) ensures ~/.tmux.conf sources this repo's tmux.conf; (3) writes and enables the systemd user service pointing at the repo's tmux.conf (skipped with --no-boot); (4) writes the desktop entry invoking cj. Regenerating the unit and desktop file from the resolved repo path makes the install work from any clone location.

cj — claude-join

A flat bash script (no functions) that attaches to the persistent claude session. Documented behaviour:

cj [claude-args…]

  • If $TMUX is set (already inside tmux): execs the real Claude CLI in the current pane — no nesting.
  • Otherwise ensures the claude session exists (fallback if the boot service hasn't started it), then reuses an idle shell window by launching Claude in it, or opens a new window if every window is busy, and attaches.

Env: CJ_CLAUDE overrides the Claude binary path (default ~/.local/bin/claude); used for testing and if the binary moves.

Does not shadow claude — that still launches a plain, non-tmux instance.

claude-tmux.service — boot autostart

A systemd user service (~/.config/systemd/user/claude-tmux.service) that starts the statically-named session at boot.

DirectiveWhy
Type=forkingtmux daemonizes the server and the launching client exits.
ExecStart=tmux -f …/tmux.conf new-session -d -s claudeCreates the server + session in the unit's cgroup.
ExecStop=tmux kill-session -t claudeClean teardown on stop.
RemainAfterExit=yesUnit stays active after the forking launcher exits.
WantedBy=default.targetEnabled to start at boot; linger keeps it across logout.

The server lives under user@1000.service/app.slice/claude-tmux.service — owned by the user manager, so it (and every Claude instance spawned inside it) is independent of any launching terminal.

Usage

claude-session frontend backend docs        # one Claude per window
claude-session api:~/src/api web:~/src/web  # set a starting dir per window
claude-session -s review pr1 pr2            # a second, separate session
ssh parrot -t 'claude-session -g'           # attach from phone, own size
